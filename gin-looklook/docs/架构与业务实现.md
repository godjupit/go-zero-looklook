# Gin 单体架构与业务实现

## 一、改造边界

原项目按 API、RPC、MQ 拆成 11 个进程。新项目把这些进程合并为一个 Gin 可执行程序，但保留四个业务域、四个数据库，以及 Redis、Kafka、Asynq、Prometheus、Jaeger、Filebeat、Elasticsearch、Kibana、Grafana、Asynqmon。这样部署和本地调试更简单，同时仍能展示领域分层、异步解耦、状态机和可观测性。

## 二、代码分层

- `cmd/server`：进程装配、双 HTTP Server、优雅退出。
- `internal/httpapi`：17 个兼容路由、参数校验、JWT、响应封装、指标和链路追踪。
- `internal/service`：用户、民宿、订单、支付用例，是业务规则的核心。
- `internal/repository`：四库数据访问、事务、Redis 缓存和乐观锁。
- `internal/worker`：Asynq 延迟/定时任务、Kafka 支付事件消费者、微信通知。
- `internal/platform`：错误码、JWT、流水号、金额转换和 OpenTelemetry 初始化。

## 三、关键业务链路

### 注册与登录

注册在同一个数据库事务里写 `user` 与 `user_auth`，任何一步失败都回滚，避免“有用户但没有认证记录”。登录保持原项目的 MD5 密码兼容逻辑，成功后签发 JWT。用户详情按 ID 读取 Redis，未命中再查询 MySQL 并回填十分钟缓存。

### 民宿查询

首页活动、商家民宿、猜你喜欢、民宿详情、优质房东、商家列表/详情和评论列表都由 TravelService 组织。详情使用 Redis cache-aside；金额在数据库和领域层以“分”保存，只在 HTTP View 转换为“元”，避免浮点累计误差。

### 创建订单

创建前读取民宿快照，校验入住时间至少一晚，然后计算住宿与餐饮总价。订单保存标题、价格、房东等快照，即使民宿后来改价，历史订单也不变。入库成功后创建 30 分钟 Asynq 延迟任务；任务触发时只关闭仍处于待支付状态的订单。

### 状态机与并发

合法流转为：待支付→取消或待使用；待使用→已使用、退款或过期。数据库更新携带 `version` 条件并原子加一，重复回调或并发消费者只有一个成功。相同目标状态按幂等成功处理。

### 微信支付与最终一致性

预支付校验订单归属和微信授权，先保存本地支付流水，再调用微信 JSAPI。回调验签、核对金额后，在同一个本地事务中以乐观锁更新流水并写入 Outbox 事件。后台投递器把事件发送到 Kafka，成功后标记已发布，失败则指数退避重试。它消除了“数据库成功、Kafka 失败”导致的永久不一致窗口。Kafka 采用至少一次语义，消费端通过状态机幂等处理重复消息。事件同时包含支付流水号和订单号，消费者按订单号推进订单状态，并通过 Asynq 发送订阅消息。

### 秒杀链路

秒杀入口通过 Redis Lua 原子完成时间校验、一人一单、库存预扣与 Stream 消息写入，避免请求洪峰直接访问 MySQL，也消除预扣库存与异步消息之间的丢失窗口。消费者在 MySQL 事务中用条件更新和唯一索引做第二层防超卖与幂等，然后生成可复用原支付流程的普通订单。确定失败时执行幂等 Lua 补偿。专项设计和并发验证见 [秒杀机制设计与实现](秒杀机制设计与实现.md)。

### 缓存击穿保护

用户和民宿详情采用 cache-aside；同一热点 Key 在缓存失效瞬间的并发回源由 `singleflight` 合并为一次 MySQL 查询，其余请求共享结果。这项保护不改变缓存语义，却能避免热点失效时数据库流量尖峰。缓存 Key 带结构版本，DTO 演进时可主动失效旧值；用户密码字段明确禁止 JSON 序列化，避免敏感凭据进入 Redis。

## 四、可观测性和可靠性

Gin 中间件记录请求数与耗时，独立 `4000` 端口暴露 Prometheus 指标；OpenTelemetry 将链路发送到 Jaeger。容器标准输出由 Filebeat 采集，经 Kafka 和 go-stash 写入 Elasticsearch，在 Kibana 查询。HTTP Server、Worker、Scheduler、Kafka Reader 在 SIGTERM 时统一优雅退出。支付 Outbox 使用唯一事件键防止重复入箱，并以指数退避限制故障期间的重试压力。

## 五、面试时应主动说明

这是“模块化单体”，适合业务规模尚未需要微服务时使用。它减少服务注册、RPC、分布式部署和排障成本，却保留清晰领域边界；未来流量或团队规模增长时，可以沿 Service/Repository 边界重新拆分。当前示例为了兼容原数据继续使用 MD5，生产系统应迁移到 bcrypt/Argon2，并配合密码升级策略。支付凭证必须来自密钥管理服务，不能写入仓库。
