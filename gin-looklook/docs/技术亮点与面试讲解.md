# 技术亮点与面试讲解

这份文档只讲已经在代码中实现并验证过的能力。面试时建议先讲业务问题，再讲方案和取舍，不要只罗列组件名。

## 一、项目定位

这是一个民宿交易系统，覆盖用户注册登录、微信授权、民宿与商家查询、评论、订单、微信支付、延迟关单、支付后通知，以及监控、链路和日志平台。原版由 go-zero API/RPC/MQ 共 11 个进程组成；新版改成 Gin 模块化单体：一个进程、四个领域模块、四个数据库，外部中间件保持不变。

面试开场可以这样说：

> 我没有把微服务简单地合成一个大 Handler，而是只合并部署边界。用户、旅行、订单、支付仍通过 Service 和 Repository 隔离，所以当前享受单体的低运维成本，将来也能沿模块边界重新拆分。

## 二、已实现的核心亮点

### 1. 模块化单体，而不是分层失控的单体

`httpapi → service → repository` 是单向依赖；Worker 只消费领域服务，不绕过业务规则直接改状态。Handler 负责 JSON 和身份，Service 负责用例，Repository 负责 SQL/Redis。这样业务规则能脱离 Gin 独立测试，也避免未来所有代码互相调用。

对应代码：`internal/httpapi`、`internal/service`、`internal/repository`、`internal/worker`。

### 2. 订单快照、整数金额与显式状态机

下单时复制民宿标题、房东、餐食和价格，防止房源改价影响历史订单。所有金额在数据库和领域层以“分”的 `int64` 保存，只在响应层换算成元。订单状态只允许：待支付→取消/待使用，待使用→已使用/退款/过期；更新 SQL 同时校验 `version`，并发回调只能有一个成功。相同目标状态按幂等成功处理。

这是面试中最值得展开的一条：快照解决历史一致性，整数金额解决浮点误差，状态机解决非法跳转，乐观锁解决并发覆盖，四者针对的是不同问题。

### 3. Transactional Outbox 解决数据库与 Kafka 双写

微信成功回调不是“先更新数据库，再直接发 Kafka”。支付状态更新和 `event_outbox` 写入在同一个 MySQL 事务提交；后台投递器扫描待发送事件，成功后标记完成，失败使用最高 60 秒的指数退避。事件键有唯一索引，避免同一状态重复入箱。

这提供至少一次投递：极端情况下可能重复发送，但不会永久丢事件。订单消费者通过状态幂等抵御重复消息。它比声称“Kafka 很可靠”更有技术含量，因为真正处理了 DB/Kafka 之间没有共同事务的问题。

对应代码：`Repository.UpdatePaymentWithOutbox`、`Runtime.publishOutbox`、`005_payment_outbox.sql`。

### 4. Asynq 延迟关单和定时任务

订单创建后写入 30 分钟延迟任务。任务执行时重新读取订单，只取消仍处于待支付状态的记录，已支付订单不会被误关。Scheduler 每分钟执行结算示例；支付成功通知也通过任务队列异步处理并重试微信调用。

需要主动说明的取舍：当前为了保持原业务行为，下单落库成功后即使队列临时失败仍向用户返回成功并记录错误。生产升级方向是把“关单任务”也纳入 Outbox/补偿扫描。

### 5. Cache-aside、击穿保护和敏感数据治理

用户详情和民宿详情使用 Redis cache-aside。缓存失效瞬间，同一 Key 的并发 MySQL 回源由 `singleflight` 合并，避免热点击穿。Key 包含结构版本 `v2`，缓存 DTO 变化时能淘汰不兼容旧值。用户密码使用 `json:"-"`，不会进入 Redis；民宿价格经过序列化往返测试，避免因字段遗漏产生零价订单。

这一点来自真实运行中发现的风险：缓存对象不是“能 Marshal 就算完成”，还要验证敏感字段和关键业务字段。

### 6. 可观测性不是一个 `/metrics`

Gin 中间件按路由记录请求量、HTTP 状态和耗时直方图；独立 `4000` 端口给 Prometheus 抓取，避免业务路由故障影响采集。OpenTelemetry 将 HTTP 链路发往 Jaeger。容器日志由 Filebeat 收集，经 Kafka、go-stash 写入 Elasticsearch，再由 Kibana 查询；Grafana 展示指标。

独立指标端口、链路、指标和集中日志形成三类互补证据：指标回答“有没有问题”，链路回答“慢在哪里”，日志回答“具体发生了什么”。

### 7. 一个进程内统一生命周期

HTTP、指标 Server、Asynq Worker、Scheduler、Kafka Reader 和 Outbox Publisher 都由根 Context 管理。SIGTERM 到来后先停止接收请求，再关闭 Worker、消息连接、数据库和追踪器。实现中使用 Asynq 的非阻塞 `Start`，避免组件和主进程同时监听系统信号造成重复 Shutdown。

### 8. 秒杀：Lua + Stream + MySQL 双层屏障

秒杀不是让请求直接竞争数据库行锁。Lua 使用 Redis 服务器时间，原子完成活动校验、一人一单、库存预扣，并在同一脚本 `XADD`，避免库存与异步消息双写。Stream Consumer Group 异步削峰，Pending 消息可被重新认领；MySQL 条件更新和两个唯一索引是最终防超卖与幂等屏障。失败补偿 Lua 只处理 `pending` 预约，重复执行不会多加库存。30 用户并发抢 20 库存的结果严格为 20 成功、10 售罄。

## 三、从原架构到新架构的取舍

| 维度 | go-zero 微服务版 | Gin 模块化单体版 |
|---|---|---|
| 部署单元 | 11 个 API/RPC/MQ 进程 | 1 个业务进程 |
| 模块调用 | gRPC | Go 方法调用 |
| 数据边界 | 四库 | 四库不变 |
| 异步能力 | Kafka + Asynq | Kafka + Asynq 不变 |
| 运维成本 | 服务发现、RPC 配置、多个指标端口 | 单进程、统一配置和生命周期 |
| 故障隔离 | 较强 | 进程级隔离较弱 |
| 适用阶段 | 团队和流量已需要独立扩缩容 | 中小团队、业务边界仍在快速变化 |

不要说“单体一定比微服务好”。更准确的回答是：在当前规模，单体降低协调和运维成本；当支付、订单需要独立扩缩容或团队自治时，再按已有模块边界拆分。

## 四、完整业务链路怎么讲

以支付成功为例：

1. JWT 中间件解析用户，预支付校验订单归属和微信授权。
2. 保存本地 `PMT` 支付流水，再调用微信 JSAPI。
3. 微信回调经过平台证书验签，并核对付款金额。
4. 乐观锁更新支付流水，同时在本地事务写 Outbox。
5. Outbox Publisher 投递 Kafka，失败指数退避。
6. Kafka 消费者按订单号将订单从待支付推进为待使用。
7. 状态更新成功后投递 Asynq 微信订阅通知任务。
8. 全过程可从 Prometheus、Jaeger 和 Kibana 观察。

这一条链路同时展示安全校验、金额一致性、并发控制、可靠事件、幂等消费、异步削峰和可观测性。

## 五、面试官可能追问

### 为什么有 Kafka 还需要 Asynq？

Kafka 用于跨模块的业务事件流和消费组扩展；Asynq 适合 Redis 驱动的延迟任务、定时任务和单任务重试。支付状态传播用 Kafka，30 分钟关单和微信通知用 Asynq，职责不同。

### Outbox 会不会重复消费？

会存在“Kafka 已成功、标记 Outbox 前进程退出”的窗口，因此设计目标是至少一次而非恰好一次。消费端必须幂等，本项目用订单当前状态与乐观锁处理重复事件。

### singleflight 能替代分布式锁吗？

不能。它只合并单个进程内的并发回源；多副本仍可能各回源一次。但每个副本最多一次，通常足以显著削峰。若单个 Key 的跨实例压力仍不可接受，可增加逻辑过期、互斥重建或热点永不过期方案。

### 为什么还保留四个数据库？

为了保留领域数据边界和将来拆分的可能性。单体事务只在单个领域库内使用；支付到订单跨库一致性通过事件完成，不伪装成一个跨库事务。

### 还有哪些生产化改进？

- 原项目为数据兼容保留 MD5，生产应渐进迁移 bcrypt/Argon2。
- 延迟关单入队可进一步改为 Outbox，消除订单/队列双写窗口。
- Outbox 多副本投递应加 `SKIP LOCKED` 认领或租约字段。
- 增加限流、熔断、OpenAPI、数据库迁移版本管理和更完整集成测试。
- 微信和 JWT 密钥接入 KMS/Secret Manager，禁止使用示例默认值。
