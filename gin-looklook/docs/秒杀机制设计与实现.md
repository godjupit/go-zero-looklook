# 秒杀机制设计与实现

## 一、目标与约束

秒杀不是普通下单接口前加一个 Redis `DECR`。本实现同时处理以下问题：

- 高并发请求不能直接打满 MySQL。
- 库存不能超卖。
- 同一用户同一活动只能成功一次，重复请求必须幂等。
- Redis 预扣成功后，异步消息不能在进程崩溃时丢失。
- 消费者重复处理不能生成重复订单。
- MySQL 落单失败要恢复 Redis 库存和用户资格。
- 客户端需要查询异步下单结果。

## 二、整体链路

```text
POST /order/v1/seckill/reserve
        |
        v
Redis Lua（时间校验 + 一人一单 + 库存预扣 + XADD）
        |
        v
Redis Stream Consumer Group
        |
        v
MySQL 事务（二次库存屏障 + 唯一索引 + 创建订单）
        |
        +--> 成功：Redis result=success，ACK Stream
        |
        +--> 失败：重试；确定售罄/达到上限后 Lua 补偿并 ACK
```

抢购接口只执行一次 Redis Lua，快速返回 `reservationSn`。客户端通过 `/order/v1/seckill/result` 查询 `pending / success / failed`，成功后拿到普通订单号，后续支付链路不需要特殊处理。

## 三、为什么使用 Redis Stream

如果 Lua 只做库存扣减，随后由 Go 调用 Kafka 或 Asynq，会存在“库存已扣，但进程在发消息前退出”的窗口。这里在同一个 Lua 脚本中同时执行库存预扣和 `XADD`，两者共享 Redis 的原子执行边界，不会出现只有库存、没有下单消息的中间状态。

Stream 使用 Consumer Group：

- 新消息由 `XREADGROUP` 分配给消费者。
- MySQL 订单提交并更新结果后才 `XACK`。
- 未确认消息由 `XAUTOCLAIM` 回收，进程重启后仍可继续处理。
- 业务失败保留 Pending，确定失败或重试达到 5 次才补偿并 ACK。

所有 Lua Key 都包含 `{seckill}` hash tag，在 Redis Cluster 中会落到同一 slot，避免多 Key Lua 的 `CROSSSLOT` 错误。

## 四、双层防超卖

### 第一层：Redis Lua

脚本使用 Redis 服务器的 `TIME`，避免应用实例时钟偏差。执行顺序是：

1. 校验活动存在、启用、已开始且未结束。
2. 在用户 Hash 中查询历史预约；存在则返回原预约号。
3. 校验库存大于零并 `DECR`。
4. 写入用户→预约号映射和预约结果 Hash。
5. `XADD` 下单消息。

Lua 返回码区分成功、重复请求、未开始、已结束、售罄和活动不存在。

预约号不使用高并发下容易发生生日碰撞的短随机数，而是使用 Redis `INCR` 全局序列；秒杀订单号由预约号确定性映射，因此消息重放时天然得到同一个订单号。

### 第二层：MySQL

Redis 是流量闸门，MySQL 是最终事实来源。异步落单事务执行：

```sql
UPDATE seckill_activity
SET sold_count = sold_count + 1
WHERE id = ? AND status = 1 AND sold_count < stock;
```

受影响行数为零即拒绝下单。`seckill_order` 还有两个唯一索引：

- `reservation_sn`：同一 Stream 消息重复消费不会重复下单。
- `(activity_id, user_id)`：即使 Redis 数据丢失，同一用户也不能在数据库创建第二笔秒杀订单。

活动库存、秒杀记录和普通订单在同一个 `looklook_order` 本地事务提交，不存在半笔订单。

## 五、失败补偿与幂等

消费者可能在“事务提交成功、Redis 结果更新前”退出。消息重新处理时，MySQL 唯一索引会返回已经生成的订单号，然后把 Redis 结果补成成功，因此不会重复扣库存。

如果 Redis 用户占位丢失、但 MySQL 中该用户已有订单，新预约会先预扣一次库存。MySQL 的 `(activity_id,user_id)` 唯一索引返回原订单号，完成结果的 Lua 会通过 `stockRestored` 标记幂等恢复这一次多余预扣。即使进程在恢复后再次处理消息，也不会把库存重复加回。

确定失败时，补偿 Lua 仅在预约状态仍为 `pending` 时执行：

1. Redis 库存加一。
2. 删除用户占位，使用户可以重试。
3. 将结果设置为 `failed` 并保存公开错误信息。

状态判断、库存恢复和用户解锁是同一个 Lua 原子操作，重复补偿不会多加库存。

## 六、活动预热和缓存恢复

进程启动时从 MySQL 加载未结束活动，将时间窗口和剩余库存写入 Redis。库存使用 `SETNX`：普通应用重启不会把已经扣减的库存重置；Redis 数据确实丢失时，则使用 `stock - sold_count` 从 MySQL 恢复。Key 在活动结束七天后过期，给异步订单和查询结果预留处理时间。

Docker Compose 中 Redis 开启 AOF，Stream、库存和 Pending 消息都具备持久化基础。

## 七、接口

### 活动列表（无需登录）

```http
POST /travel/v1/seckill/activityList
Content-Type: application/json

{}
```

返回秒杀价、总库存、Redis 实时剩余库存和活动时间。

### 抢购（需要 JWT）

```http
POST /order/v1/seckill/reserve
Authorization: Bearer <token>
Content-Type: application/json

{
  "activityId": 1,
  "liveStartTime": 1785600000,
  "liveEndTime": 1785772800,
  "livePeopleNum": 2,
  "remark": "seckill"
}
```

成功受理返回 `reservationSn` 和当前状态。重复提交返回同一个预约号。

### 查询结果（需要 JWT）

```http
POST /order/v1/seckill/result
Authorization: Bearer <token>
Content-Type: application/json

{"reservationSn":"SKR..."}
```

只有预约所属用户可以查询。`success` 时返回 `orderSn`，可继续走原微信支付接口。

## 八、监控指标

- `gin_looklook_seckill_reservations_total{result=...}`：受理、售罄、未开始、已结束等入口结果。
- `gin_looklook_seckill_orders_total{result=...}`：异步订单成功和失败尝试。
- Stream 的 Pending 数量用于观察消费堆积。
- MySQL `stock - sold_count` 与 Redis stock 可用于库存对账告警。

## 九、真实并发验证

本地使用 30 个不同 JWT 用户并发抢 20 份库存，结果如下：

| 检查项 | 结果 |
|---|---:|
| HTTP 受理成功 | 20 |
| 售罄响应 | 10 |
| Redis 剩余库存 | 0 |
| MySQL sold_count | 20 |
| 秒杀订单数 | 20 |
| 不同用户数 | 20 |
| 不同预约号 | 20 |
| 不同订单号 | 20 |
| Stream Pending | 0 |

另外验证了：同一用户重复请求返回相同预约号且订单数不变；人为制造 Redis/MySQL 库存不一致后，MySQL 屏障阻止下单，补偿 Lua 将 Redis 库存恢复并删除用户占位。

## 十、面试时主动说明的边界

- 当前使用单个全局 Stream，适合中等规模；极大流量可以按活动分片 Stream 和消费组。
- Redis Cluster 为保证 Lua 原子性将秒杀 Key 固定到同一 slot；超大规模可按活动分组 hash tag 分片。
- 活动管理后台、限流验证码和防机器人不是交易正确性的核心，本项目暂未实现。
- 生产环境还应增加库存定时对账、死信人工处理、压测基线和容量预估。
